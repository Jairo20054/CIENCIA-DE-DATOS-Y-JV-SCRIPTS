const / let

Esencial

Qué es: const para valores que no se reasignan, let para los que sí. Nunca uses var — tiene hoisting y scope impredecible que genera bugs sutiles.

✗ Anti-patrón 
// var tiene hoisting y scope de función
var price = 1500000;
var price = 2000000; // sin error — redeclaración

// let para valores que no cambian
let API_URL = "https://api.nido.co";
✓ Buena práctica 
// const por defecto — solo let cuando mutes
const MAX_PRICE = 50_000_000;
const API_URL = "https://api.nido.co";

let currentPage = 1;
currentPage += 1; // reasignación válida
Tip senior: Usa const por defecto. Solo cambia a let cuando necesites reasignar. Esto comunica intención al equipo y evita mutaciones accidentales. Usa separador de miles _ para números grandes.
Mini ejercicio: Declara MAX_RESULTS (constante), currentPage (mutable) y BASE_URL (constante). Intenta reasignar MAX_RESULTS y observa el error.

Arrow functions

Esencial

Qué es: forma moderna de funciones. Se usa en callbacks, eventos, map, filter y componentes React. No tiene su propio this.

✗ Anti-patrón 
// Función sin tipo de retorno claro
const calc = (a, b, c) => {
  return a * b + c - a / b;
};

// Nombre no descriptivo
const fn1 = (x) => x * 1.19;
✓ Buena práctica 
// Nombre descriptivo y una sola responsabilidad
const addIVA = (price) => price * 1.19;

const formatCOP = (price) =>
  `$${price.toLocaleString("es-CO")}`;

// Explícito: qué entra y qué sale
const getFullName = (first, last) =>
  `${first} ${last}`;
Tip senior: Una función debe hacer UNA cosa. Si necesitas scroll para leerla, es hora de refactorizar. El nombre debe ser un verbo que describa su acción: formatPrice, validateEmail, calculateTotal.
Mini ejercicio: crea addIVA(price), formatCOP(price) y getPriceWithTax(price) que componga las dos anteriores.

Async/Await + Error handling

Esencial

Qué es: maneja operaciones que tardan (API, BD). await espera el resultado y try/catch previene que el programa se rompa.

✗ Anti-patrón 
// Sin manejo de errores
const getProperties = async () => {
  const res = await fetch("/api/properties");
  return res.json(); // si falla → app rota
};
✓ Buena práctica 
const getProperties = async (cityId) => {
  try {
    const res = await fetch(
      `/api/properties?city=${cityId}`
    );
    if (!res.ok) {
      throw new Error(`HTTP ${res.status}`);
    }
    return await res.json();
  } catch (error) {
    console.error("Fetch failed:", error);
    return [];
  }
};
Tip senior: Siempre verifica res.ok — un status 404 o 500 no lanza excepción por sí solo. Retorna un valor seguro en el catch (array vacío, null, objeto con error) para que la UI no se rompa.
Mini ejercicio: escribe getUsers() que consulte /api/users, valide el status, retorne [] si falla, y logee el error con contexto.

Destructuring

Esencial

Qué es: extrae valores de objetos o arrays en una línea. En React es fundamental para leer props, state y respuestas de API.

// Objetos
const { title, city, price } = property;

// Con valores por defecto — defensivo
const { title, city = "Sin ciudad", price = 0 } = property;

// Renombrar variables
const { title: propertyTitle, landlord: { name: ownerName } } = property;

// Arrays
const [first, second, ...rest] = results;
Tip senior: Usa valores por defecto en destructuring para datos que vienen del servidor — nunca confíes en que un campo existirá. Destructuring anidado profundo reduce legibilidad; extrae en pasos.
Mini ejercicio: de un objeto user extrae name, email, y role con default "viewer".

.map(), .filter(), .reduce()

Esencial

Qué es: map transforma cada elemento, filter selecciona, reduce acumula. Son inmutables: nunca modifican el array original.

const properties = [
  { city: "Cali", price: 900_000, available: true },
  { city: "Bogotá", price: 2_000_000, available: false },
  { city: "Cali", price: 1_300_000, available: true },
];

// filter → solo las que cumplen condición
const available = properties.filter((p) => p.available);

// map → transformar cada elemento
const titles = properties.map((p) => p.city);

// Encadenar — filter + map
const caliPrices = properties
  .filter((p) => p.city === "Cali" && p.available)
  .map((p) => formatCOP(p.price));

// reduce — calcular total
const totalPrice = properties.reduce(
  (sum, p) => sum + p.price, 0
);
Tip senior: Nunca mutes el array original con push, splice, o sort() directo. Usa [...arr].sort() o toSorted(). Encadenar filter+map es más legible que un reduce complejo. Si el reduce tiene más de 3 líneas, probablemente necesitas refactorizar.
Mini ejercicio: filtra propiedades de Cali con precio < $1.500.000, muestra solo título y precio formateado.

Spread / Rest operator

Esencial

Qué es: ... copia arrays u objetos sin mutarlos. En React es obligatorio para actualizar estado de forma segura.

// Copiar y modificar objeto (inmutable)
const updated = { ...property, price: 1_800_000 };

// Agregar a array sin mutar
const withNew = [...properties, newProperty];

// Eliminar de array sin mutar
const without = properties.filter((p) => p.id !== removeId);

// Rest en parámetros — recoger "el resto"
const logFirst = (first, ...others) => {
  console.log(first, others.length);
};
Cuidado: Spread hace copia superficial. Si tienes objetos anidados, necesitas copiar cada nivel o usar structuredClone() para copia profunda.
Mini ejercicio: copia un objeto property cambiando solo el precio. Luego agrega una nueva propiedad a un array sin mutar el original.

Optional chaining ?. y ??

Recomendado

Qué es: ?. evita errores al acceder propiedades que podrían ser null. ?? da valor por defecto solo para null/undefined (no para 0 o "").

✗ Anti-patrón 
// || trata 0, "" y false como falsy
const rooms = property.bedrooms || 1;
// Si bedrooms es 0 → devuelve 1 (bug)

// Sin optional chaining
const name = property && property.landlord
  && property.landlord.name;
✓ Buena práctica 
// ?? solo para null/undefined
const rooms = property.bedrooms ?? 1;
// Si bedrooms es 0 → devuelve 0 ✓

// Acceso seguro anidado
const name = property?.landlord?.name
  ?? "Sin propietario";
Mini ejercicio: muestra el nombre del usuario si existe, si no muestra "Invitado". Usa ?? para un precio que podría ser 0.

import / export (ESM)

Recomendado

Qué es: ES Modules permiten separar tu código en archivos. Un componente, hook, servicio o utilidad debe poder exportarse e importarse donde se necesite.

// utils/formatPrice.ts — named export
export const formatPrice = (price: number): string =>
  `$${price.toLocaleString("es-CO")}`;

// PropertyCard.tsx — import nombrado
import { formatPrice } from "../utils/formatPrice";

// Barril de exportaciones (index.ts)
export { formatPrice } from "./formatPrice";
export { formatDate } from "./formatDate";
Tip senior: Prefiere named exports sobre default exports — facilita el refactoring y el auto-import. Usa barrel files (index.ts) para agrupar exportaciones de un directorio. Evita barrel files en directorios muy grandes — puede afectar tree-shaking.

Closures y Event Loop

Avanzado

Qué es: un closure es una función que "recuerda" su contexto externo. El event loop explica cómo JS maneja asincronía siendo single-thread. Son la base mental para hooks, timers, promesas y Node.

// Closure: la función interna recuerda count
const createCounter = (initial = 0) => {
  let count = initial;
  return {
    increment: () => ++count,
    decrement: () => --count,
    getCount: () => count,
  };
};

const counter = createCounter(10);
counter.increment(); // 11
counter.getCount(); // 11

// Event loop: microtasks vs macrotasks
console.log("1");                    // sync
setTimeout(() => console.log("2"), 0); // macro
Promise.resolve().then(() =>
  console.log("3"));                   // micro
// Output: 1, 3, 2
Tip senior: Los closures son la base de useState en React — cada render "captura" su propio estado. Entender el event loop te permite debuggear race conditions y entender por qué setTimeout(0) no ejecuta inmediatamente.

Componente funcional + JSX

Esencial

Qué es: un componente es una función que retorna JSX. Con TypeScript, tipas las props para auto-completado y seguridad.

✗ Anti-patrón 
// Props sin tipar, nombre genérico
function Card(props) {
  return (
    <div>
      <h3>{props.data.title}</h3>
      <p>{props.data.city}</p>
    </div>
  );
}
✓ Buena práctica 
interface PropertyCardProps {
  title: string;
  city: string;
  price: number;
}

function PropertyCard({
  title, city, price
}: PropertyCardProps) {
  return (
    <article className="card">
      <h3>{title}</h3>
      <p>{city} · {formatCOP(price)}</p>
    </article>
  );
}
Tip senior: Usa destructuring directo en los parámetros. Tipa las props con interface. Usa article, section, nav en vez de div para HTML semántico.

useState

Esencial

Qué es: guarda estado local. Cuando el estado cambia, React re-renderiza el componente.

// Estado simple
const [isOpen, setIsOpen] = useState(false);

// Estado de objeto — SIEMPRE con spread
const [filters, setFilters] = useState<Filters>({
  city: "",
  maxPrice: null,
  bedrooms: null,
});

// Actualizar con función — usa prev, nunca el estado actual
const updateCity = (city: string) => {
  setFilters((prev) => ({ ...prev, city }));
};
Cuidado: setState es asíncrono. No leas el estado justo después de setearlo — usa la forma funcional setState(prev => ...) para actualizar basado en el valor anterior.

useEffect

Esencial

Qué es: ejecuta efectos secundarios: cargar datos, suscripciones, timers. El array de dependencias controla cuándo corre.

✗ Anti-patrón 
// Sin cleanup — memory leak
useEffect(() => {
  const id = setInterval(poll, 5000);
  // falta return () => clearInterval(id)
}, []);

// Dependencia faltante
useEffect(() => {
  fetchData(userId);
}, []); // userId falta — bug sutil
✓ Buena práctica 
useEffect(() => {
  const controller = new AbortController();

  const loadProperties = async () => {
    try {
      const data = await fetchProperties(
        city, { signal: controller.signal }
      );
      setProperties(data);
    } catch (err) {
      if (err.name !== "AbortError") {
        setError(err.message);
      }
    }
  };

  loadProperties();
  return () => controller.abort();
}, [city]); // dependencia correcta
Tip senior: Siempre incluye cleanup con AbortController para peticiones. Usa el linter de exhaustive-deps. Si tu useEffect es complejo, probablemente necesitas un custom hook.

Renderizado condicional

Esencial

Qué es: muestra UI diferente según el estado: loading, error, vacío, datos. Es el patrón más usado en React.

function PropertyList({ city }: { city: string }) {
  const { properties, isLoading, error } = useProperties(city);

  if (isLoading) return <Skeleton count={6} />;
  if (error) return <ErrorBanner message={error} />;
  if (properties.length === 0) return <EmptyState />;

  return (
    <section className="grid">
      {properties.map((p) => (
        <PropertyCard key={p.id} {...p} />
      ))}
    </section>
  );
}
Tip senior: Usa early returns (guard clauses) en vez de ternarios anidados — es mucho más legible. Siempre maneja los 4 estados: loading, error, vacío y datos.

Custom Hooks

Recomendado

Qué es: funciones que empiezan con use y encapsulan lógica reutilizable. Separan la lógica de la UI.

// hooks/useProperties.ts
export function useProperties(city: string) {
  const [properties, setProperties] = useState<Property[]>([]);
  const [isLoading, setIsLoading] = useState(true);
  const [error, setError] = useState<string | null>(null);

  useEffect(() => {
    const controller = new AbortController();
    setIsLoading(true);
    setError(null);

    propertyService.getByCity(city, controller.signal)
      .then(setProperties)
      .catch((err) => {
        if (err.name !== "AbortError") setError(err.message);
      })
      .finally(() => setIsLoading(false));

    return () => controller.abort();
  }, [city]);

  return { properties, isLoading, error };
}
Tip senior: Un custom hook debe seguir SRP: una razón para existir. El componente queda limpio y testeable. El hook queda reutilizable en otra pantalla.

REST API y códigos HTTP

Esencial

Qué es: una API REST usa HTTP para CRUD. Las URLs deben ser sustantivos en plural. Los status codes comunican el resultado.

// Verbos + sustantivos = API REST correcta
GET    /api/properties          // listar
GET    /api/properties/:id      // detalle
POST   /api/properties          // crear
PATCH  /api/properties/:id      // actualizar parcial
DELETE /api/properties/:id      // eliminar

// Status codes que debes conocer
200 OK            // éxito
201 Created       // recurso creado
400 Bad Request   // datos inválidos del cliente
401 Unauthorized  // sin autenticación
403 Forbidden     // sin permisos
404 Not Found     // recurso no existe
409 Conflict      // duplicado o conflicto
500 Internal      // error del servidor
Tip senior: usa PATCH en vez de PUT para actualizaciones parciales. Nunca expongas errores internos al cliente — loggea el stack trace y responde un mensaje genérico.

Middleware

Esencial

Qué es: función intermedia entre petición y respuesta. Sirve para auth, logs, validación, rate-limiting y manejo de errores.

// Auth middleware con manejo de errores
const requireAuth = async (req, res, next) => {
  try {
    const token = req.headers.authorization
      ?.replace("Bearer ", "");

    if (!token) {
      return res.status(401).json({
        error: "TOKEN_MISSING",
        message: "Token de autenticación requerido",
      });
    }

    const payload = verifyToken(token);
    req.user = payload; // disponible en controllers
    next();
  } catch {
    res.status(401).json({
      error: "TOKEN_INVALID",
      message: "Token inválido o expirado",
    });
  }
};

Capas: Route → Controller → Service

Recomendado

Qué es: separa responsabilidades. La ruta define URL, el controller maneja HTTP, el service contiene la lógica de negocio.

// routes/property.routes.ts
router.get("/", listProperties);
router.post("/", requireAuth, validate(createSchema), createProperty);

// controllers/property.controller.ts
export const listProperties = async (req, res, next) => {
  try {
    const filters = parseFilters(req.query);
    const result = await propertyService.list(filters);
    res.json(result);
  } catch (error) {
    next(error); // delega al error handler global
  }
};

// services/property.service.ts
export const list = async (filters: PropertyFilters) => {
  const [results, total] = await Promise.all([
    prisma.property.findMany({ where: buildWhere(filters) }),
    prisma.property.count({ where: buildWhere(filters) }),
  ]);
  return { results, total };
};
Tip senior: El controller NUNCA debe importar Prisma directamente. El service NUNCA debe recibir req ni res. Esta separación permite testear cada capa independientemente.

Validación con Zod

Recomendado

Qué es: Zod valida y tipa datos del cliente en una sola declaración. Nunca confíes en el frontend.

import { z } from "zod";

Schema, modelos y relaciones

Esencial 
model Property {
  id         Int       @id @default(autoincrement())
  title      String
  price      Int
  city       String
  bedrooms   Int       @default(1)
  available  Boolean   @default(true)
  createdAt  DateTime  @default(now())
  updatedAt  DateTime  @updatedAt
  deletedAt  DateTime? // soft delete

  landlord   User      @relation(fields: [landlordId], references: [id])
  landlordId Int
  favorites  Favorite[]

  @@index([city])
  @@index([price])
  @@index([landlordId])
}

model Favorite {
  id         Int      @id @default(autoincrement())
  user       User     @relation(fields: [userId], references: [id])
  userId     Int
  property   Property @relation(fields: [propertyId], references: [id])
  propertyId Int
  createdAt  DateTime @default(now())

  @@unique([userId, propertyId]) // evitar duplicados

Tip senior: Siempre incluye createdAt/updatedAt. Usa soft delete (deletedAt) en producción — datos borrados en hard delete no se recuperan. Agrega índices en campos que usas en where y orderBy.

Consultas avanzadas con filtros

Esencial 
// Búsqueda con filtros opcionales + paginación
const searchProperties = async (filters: Filters) => {
  const { city, maxPrice, bedrooms, page = 1, limit = 20 } = filters;

  const where: Prisma.PropertyWhereInput = {
    deletedAt: null,        // excluir soft-deleted
    available: true,
    ...(city && { city }),
    ...(maxPrice && { price: { lte: maxPrice } }),
    ...(bedrooms && { bedrooms: { gte: bedrooms } }),
  };

  const [results, total] = await Promise.all([
    prisma.property.findMany({
      where,
      select: {
        id: true, title: true, price: true,
        city: true, bedrooms: true,
        landlord: { select: { name: true } },
      },
      orderBy: { createdAt: "desc" },
      skip: (page - 1) * limit,
      take: limit,
    }),
    prisma.property.count({ where }),
  ]);

  return {
    results,
    total,
    page,
    totalPages: Math.ceil(total / limit),
  };
};
Tip senior: Usa select para evitar traer datos innecesarios (especialmente contraseñas). Ejecuta findMany y count en paralelo con Promise.all. Construye el where dinámicamente con spread condicional.

TiposInterfacesGenerics

Tipos básicos e interfaces

Esencial

Qué es: TypeScript añade tipos estáticos a JavaScript. Detecta errores en compile-time, no en producción a las 3 AM.

// Interface para entidades del dominio
interface Property {
  id: number;
  title: string;
  city: string;
  price: number;
  bedrooms: number;
  available: boolean;
  landlord: Pick<User, "id" | "name">;
  createdAt: string;
}

// Type para filtros opcionales
type PropertyFilters = {
  city?: string;
  maxPrice?: number;
  bedrooms?: number;
  page?: number;
};

// Función tipada
const formatCOP = (price: number): string =>
  `$${price.toLocaleString("es-CO")}`;

// Union types para estados
type Status = "idle" | "loading" | "success" | "error";

// Generics — reutilizable
interface PaginatedResponse<T> {
  results: T[];
  total: number;
  page: number;
  totalPages: number;
}
Tip senior: Usa interface para objetos del dominio y type para unions, intersections y utilidades. Evita any — si no sabes el tipo, usa unknown y haz narrowing. Pick, Omit, Partial y Required son utility types esenciales.

Reglas TS para seniors

Crítico
✗ TypeScript roto 
// any desactiva TypeScript
const getData = (x: any): any => x.data;

// Type assertion sin verificar
const user = response as User;

// @ts-ignore esconde bugs
// @ts-ignore
user.nonExistentMethod();
✓ TypeScript seguro 
// unknown + narrowing
const parseResponse = (data: unknown) => {
  if (!isUser(data)) throw new Error("Invalid");
  return data; // tipado como User
};

// Zod para validar + tipar en runtime
const parsed = userSchema.parse(response);
// parsed tiene tipo inferido ✓
Regla de oro: Si tu proyecto tiene más de 5 any, tu TypeScript es JavaScript disfrazado. Usa strict: true en tsconfig. Nunca uses @ts-ignore — siempre hay una forma tipada de resolver el problema.

8. Testing

El código sin tests es código que se rompe cuando no estás mirando. Tests dan confianza para refactorizar.

VitestRTLTDD

Tests unitarios con Vitest

Esencial

Qué es: verificar que una función o módulo hace lo que promete. Si no puedes testearla fácilmente, probablemente tiene demasiadas responsabilidades.

// formatPrice.test.ts
import { describe, it, expect } from "vitest";
import { formatCOP } from "./formatPrice";

describe("formatCOP", () => {
  it("formats a positive number with COP format", () => {
    expect(formatCOP(1_500_000)).toBe("$1.500.000");
  });

  it("handles zero", () => {
    expect(formatCOP(0)).toBe("$0");
  });

  it("returns fallback for null/undefined", () => {
    expect(formatCOP(null)).toBe("Precio a consultar");
  });
});

// Patrón AAA: Arrange → Act → Assert
it("filters available properties in Cali", () => {
  // Arrange
  const properties = [
    { city: "Cali", available: true },
    { city: "Bogotá", available: true },
    { city: "Cali", available: false },
  ];
  // Act
  const result = filterByCity(properties, "Cali");
  // Assert
  expect(result).toHaveLength(1);
  expect(result[0].city).toBe("Cali");
});
Tip senior: Testea el comportamiento, no la implementación. Nombres de test descriptivos: "should [acción] when [condición]". Patrón AAA (Arrange-Act-Assert) para estructura clara. Testea edge cases: null, undefined, arrays vacíos, strings vacíos.

Testing de componentes (RTL)

Recomendado 
import { render, screen } from "@testing-library/react";
import { PropertyCard } from "./PropertyCard";

it("renders title and city", () => {
  render(<PropertyCard
    title="Casa en Cali"
    city="Cali"
    price={1500000}
  />);

  expect(screen.getByText("Casa en Cali")).toBeInTheDocument();
  expect(screen.getByText(/cali/i)).toBeInTheDocument();
});

it("shows empty state when no results", () => {
  render(<PropertyList properties={[]} />);
  expect(screen.getByText(/sin resultados/i)).toBeInTheDocument();
});
Tip senior: Testea como el usuario ve la UI — por texto, rol, label. No testees estilos ni detalles de implementación. Si necesitas data-testid, probablemente tu UI no es accesible.

9. Seguridad & Performance

Un senior no solo escribe features — protege a los usuarios y optimiza la experiencia.

XSSInjectionOptimización

Vulnerabilidades comunes

Crítico 
// ✗ SQL Injection — NUNCA concatenes queries
const query = `SELECT * FROM users WHERE email = '${email}'`;

const user = await prisma.user.findUnique({
  where: { email } // sanitizado automáticamente

// ✗ XSS — insertar HTML sin sanitizar
element.innerHTML = userInput; // PELIGROSO

Reglas de seguridad inquebrantables: 1) Nunca confíes en input del cliente — valida todo en el backend. 2) Nunca expongas stack traces en producción. 3) Hashea contraseñas con bcrypt (cost ≥ 12). 4) Usa HTTPS siempre. 5) Tokens JWT con expiración corta. 6) Rate limiting en endpoints de auth.

Arquitectura y patrones

Cómo organizar un proyecto real para que escale sin volverse un espagueti inmantenible.

Estructura de carpetas (Backend)

src/
├── routes/          // URLs y middleware por recurso
│   └── property.routes.ts
├── controllers/     // req/res — sin lógica de negocio
│   └── property.controller.ts
├── services/        // lógica de negocio — testeable
│   └── property.service.ts
├── repositories/    // acceso a BD — Prisma
│   └── property.repository.ts
├── schemas/         // validación con Zod
│   └── property.schema.ts
├── middlewares/     // auth, validate, errorHandler
├── types/           // interfaces compartidas
├── utils/           // helpers puros
├── config/          // env, DB, logger
└── app.ts           // setup Express

Estructura de carpetas (Frontend)

src/
├── components/      // UI reutilizable
│   ├── ui/          // Button, Input, Card...
│   └── property/    // PropertyCard, PropertyGrid
├── pages/           // vistas completas
│   ├── Home.tsx
│   └── PropertyDetail.tsx
├── hooks/           // custom hooks
│   └── useProperties.ts
├── services/        // llamadas a API
│   └── propertyService.ts
├── types/           // interfaces TypeScript
├── utils/           // formatPrice, formatDate...
├── context/         // AuthContext, ThemeContext
├── styles/          // CSS Modules o variables
└── App.tsx

Principios SOLID simplificados

// S — Single Responsibility
// Un módulo tiene UNA razón para cambiar
// PropertyCard renderiza UI, useProperties maneja datos

// O — Open/Closed
// Abierto para extensión, cerrado para modificación
// Agrega nuevos filtros sin modificar la función de búsqueda existente

// L — Liskov Substitution
// Un componente hijo debe funcionar donde funciona el padre

// I — Interface Segregation
// No obligues a un componente a depender de props que no usa
// ✗ <Card property={fullPropertyObject} />
// ✓ <Card title={title} price={price} city={city} />

// D — Dependency Inversion
// El service no depende de Prisma directamente → depende de una interfaz
// Esto permite cambiar de Prisma a otro ORM sin tocar la lógica
Tip senior: No necesitas aplicar SOLID a rajatabla en cada línea. Pero conocer estos principios te ayuda a tomar decisiones de diseño cuando el proyecto crece. El principio más importante en la práctica diaria: Single Responsibility.
